LXON — Documentation

A serialization format that remembers what your data is.

This is the homepage for the LXON docs. Everything else — every value, key, and container type — lives one page deeper, organized by the categories on the left. Scroll around, both panels move independently.

01 — Motivation

Why a typed format

Most text-based formats throw away type information the moment they touch the wire. A number becomes indistinguishable from a numeric string, a missing field looks the same as an explicit null, and binary data has to be smuggled through as base64. Decoders end up guessing, and every guess is a place a bug can hide.

LXON keeps the type tag attached to the value all the way through. A decoder never has to infer whether "42" was meant to be a string or a number — the format already told it.

02 — Orientation

How these docs work

The sidebar is split into three families. Value types are the atomic building blocks — string, number, boolean, null, binary, timestamp. Key types describe how a field gets named inside a container. Container types are how values get grouped — array, map, set, struct, table — each with different rules about ordering and uniqueness.

Reference pages at the bottom cover the binary encoding itself, schema validation, and how the format handles versioning over time.

03 — Design

Three rules the format doesn't break

typed always

Every value carries its type tag on the wire. There is no untyped value in LXON, only ones you haven't looked at yet.

homogeneous by default

Arrays and sets assume one value type unless you reach for a struct, which is the container built for mixed fields.

boring encoding

No clever bit-packing tricks. The byte layout is predictable enough to hand-decode with a hex viewer if you had to.

04 — A quick look

What a struct looks like

A struct is the container you reach for when fields have different types. Each field gets a key, a type, and a slot in a fixed layout:

// placeholder syntax, not final
user :: Struct {
  0x01 name    -> String
  0x02 age     -> Number
  0x03 tags    -> Set<String>
  0x04 active  -> Boolean
}

The full breakdown of key numbering, field ordering, and optional fields lives on the Struct page under Container types.

05 — Stability

Versioning & stability

Schemas are expected to change. LXON treats adding a new field as safe by default, and reserves key numbers so removed fields don't get silently reused by something else later.

The Versioning reference page goes through what counts as a breaking change and how older readers behave when they meet a newer schema.
