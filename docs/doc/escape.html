Justification

In many cases, data can't be represented as is because the parser is expecting a certain character. This is the reality in a lot of examples in programming.

To combat this, escape sequences were invented. This problem is also faced in LXON, and the solution is the same implementation as everywhere.

Escape sequences are used when you want to include a character that would otherwise be impossible to include, whether it be as a typed key or as a value. For example, if you want to write a string with a newline, you would get this: 

{
text: "Hello
World

However, this would simple parse into the following object (read !! other.comment ! comment !): 

{
text: "Hello
World: _
# This ^ symbol means undefined

In order to parse "Hello" and "World", seperated by a newline, you would need to use the escape character\n. 

{
text: "Hello\nWorld

This is incredibly useful in countless instances. Scroll to the bottom of the page to see all !! other.escape.usage ! examples !, including less obvious ones.

Syntax

To escape a character, prepend it with\. The parser will ignore the backslash and read only the character following it. If it's a code character, it will replace the pair with the represented character (\n -> newline, \s -> space, etc.), otherwise it will read the second character litterally (\" -> ").

Supported Characters

LXON offers support for many of the conventional escape characters, as well as a select assembly of custom ones. Here's all supported escape sequence codes:

  • Newline\n
  • Carriage Return\r
  • Tabulator (tab)\t
  • Space\s

If a escape sequence isn't a valid code, it will parse into the character after the backslash\, completely ignoring the backslash as is. By consequence, many commonly used escape characters aren't explicit implementations and merely an intended consequence. Here's all of the commonly used, and arguably the only other useful escape sequences:

  • Backslash\\
  • Colon\:
  • Backtick\`
  • Caret\^
  • At\@
  • Asterisk\*
  • Question Mark\?

Supported Environments

Before escaping a character, first ensure escaped characters are even supported in the current environment. For example, !! keys.regular ! regular keys ! used in !! containers.obj ! objects ! do not support escape characters at all. In the event where you wish to include unsupported characters (such as newline, colon, etc.) in the key(*see syntax rules), you must use !! cont.maps ! maps ! with string keys instead.

Here below is a full list of supported environments for escape sequences:

  • !! val.text ! Text Values !
    • !! val.text.full ! Full String !
    • !! val.text.str ! String !
    • !! val.text.char ! Char !
  • !! keys.text ! Text Keys !
    • !! keys.text.str ! String Key !
    • !! keys.text.char ! Char Key !

Usage

Here's some example usage of escape sequences as reference (read all comments, keys and values): 

{
newline: '\n
multiline: "Hello\nWorld

backtick inside: `\``

map:("
	1 colon \: inside: "val
	2 spaces at end  : "val
	\s-space at start: "val

	`caret(\^)caret^dot(.)dot
